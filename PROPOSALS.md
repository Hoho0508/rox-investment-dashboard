# Proposals

以下項目只完成分析，尚未實作。依自主優化規則，必須由專案擁有者核准後才能進入開發。

## 產業分類與自選股標籤

- **功能名稱：** 產業分類與自選股標籤
- **使用者價值：** 可依 AI、IC 設計、記憶體、被動元件、防守、軍工、能源等標籤整理與篩選追蹤股票。
- **為什麼現在需要：** 現有自選股只支援新增與刪除，資料庫沒有分類、狀態或自訂標籤欄位；需求清單中的分類功能尚未完成。
- **預估修改範圍：** Prisma migration、分類與標籤 domain module、API schema、股票清單篩選 UI、單元與 E2E 測試。
- **需要的新服務或 API：** 不需要；可先由使用者手動分類。若要自動分類才需要可信的產業資料來源。
- **可能成本：** 無新增服務費；約一個中型模組的開發與資料遷移成本。
- **安全與隱私風險：** 標籤可能暴露使用者的投資偏好；API 必須維持單人登入保護，migration 不可破壞既有自選股。
- **對既有功能的影響：** 擴充 `WatchlistItem`，需確保舊紀錄自動套用安全預設值。
- **建議優先級：** P2。
- **是否建議現在做：** 建議下一輪先確認資料模型與分類名稱，再一次完成此單一模組。

## 跨實例 API 頻率限制

- **功能名稱：** 受保護 API 的持久化頻率限制
- **使用者價值：** 避免登入、搜尋、行情與手動產報 endpoint 被重複呼叫，降低外部 API 額度消耗。
- **為什麼現在需要：** 現有行情前端有 30 秒刷新節奏，部分 Provider 有快取，但 serverless 多實例間沒有一致的 rate limit。
- **預估修改範圍：** rate-limit domain module、資料表或外部 KV adapter、API middleware、429 回應、測試與監控文件。
- **需要的新服務或 API：** 可使用現有 PostgreSQL；若改用 Redis／KV 則需另行核准服務。
- **可能成本：** PostgreSQL 方案會增加少量寫入；外部 KV 可能產生方案費用。
- **安全與隱私風險：** 不應保存完整 IP；需雜湊識別並設定短期到期時間，避免建立不必要的使用者追蹤資料。
- **對既有功能的影響：** 高頻更新可能收到 429，前端需尊重 `Retry-After` 並停止重試風暴。
- **建議優先級：** P1。
- **是否建議現在做：** 建議先採現有 PostgreSQL 的低頻關鍵 endpoint 版本；未核准前不實作。

## 盤中 WebSocket 串流

- **功能名稱：** Fugle WebSocket 盤中串流
- **使用者價值：** 比固定輪詢更即時地顯示 Tick 與 1 分 K 更新。
- **為什麼現在需要：** 現有 1 分 K 採 HTTP 請求，Tick 明確標示尚未支援。
- **預估修改範圍：** 串流 adapter、連線生命週期、斷線重連、時序儲存、前端訂閱與高負載測試。
- **需要的新服務或 API：** Fugle 串流權限，可能另需持續運作的 worker 與時序儲存。
- **可能成本：** 可能增加行情方案、運算與儲存費用。
- **安全與隱私風險：** API Key 必須只留在 server side；需防止無限重連與未授權資料轉送。
- **對既有功能的影響：** 會改變行情更新架構，必須保留目前 HTTP Provider 作為安全 fallback。
- **建議優先級：** P4。
- **是否建議現在做：** 不建議現在做；先完成分類、資料完整度與 API 頻率限制。
